## Bug 1：回复消息两条重复

### 现象
用户发送一条消息，零回复了两条完全相同的消息。

### 根因分析（二选一或两者并存）

**可能原因 A**：SSE 流结束时，`done` 事件触发了 `finishStreaming`，同时流读取循环结束后又触发了一次 `finishStreaming`。

**可能原因 B**：`handle_message` 在 `run_in_executor` 中执行后，返回结果被重复处理（线程池回调 + 主流程各处理一次）。

### 修复方案

**server.py 改动**：
找到 ChatStreamHandler 中 SSE 发送 done 事件的代码。在发送 `{"type": "done"}` 后，立即设置一个标志 `self._done_sent = True`。在流结束的其他路径中，检查这个标志，如果已发送则跳过。

```python
# 在 ChatStreamHandler 的 __init__ 或 get 方法开始时：
self._done_sent = False

# 在发送 done 事件的位置：
if not self._done_sent:
    self._done_sent = True
    await self._send({'type': 'done'})
```

**app.js 改动**：
检查 `finishStreaming` 的调用路径。在 SSE 流的 `processLine` 中处理 `done` 事件后，以及在流读取循环结束后，确保不会重复调用。

找到 `send()` 函数中的流读取循环（约第 504-519 行），在 `done` 事件的 case 中设置一个局部标志：

```javascript
let streamFinished = false;

// 在 processLine 的 done 事件中：
if (d.type === 'done') {
  if (streamFinished) return;  // 防止重复
  streamFinished = true;
  clearTimeout(dataTimeout);
  clearTimeout(timeoutId);
  finishStreaming(accumulatedReply, conv);
  // ...
}

// 在流读取循环结束后（约第 514-519 行）：
if (accumulatedReply && !streamFinished) {
  streamFinished = true;
  finishStreaming(accumulatedReply, conv);
  // ...
}
```

## Bug 2：图片上传不显示，零也没收到

### 现象
用户通过附件按钮选择图片上传后，图片没有在消息中显示，零的回复中也看不到图片内容。

### 根因分析
前端上传文件后，`uploadPendingFile` 返回了文件名，但这个文件名没有正确拼接到发送给 SSE 或 POST 的消息文本中。

### 修复方案

**app.js 改动**：
找到 `send()` 函数中上传文件后的逻辑（约第 418-422 行）：

```javascript
// 上传文件
let fName = null;
if (pendingFile) {
  fName = await uploadPendingFile();
}
```

在这之后，需要把文件名拼接到消息文本中，让 LLM 知道用户上传了文件：

```javascript
// 在 fName 获取之后，拼接文件信息到消息文本
let messageText = text;
if (fName) {
  // 判断文件类型
  const isImage = /\.(png|jpg|jpeg|gif|webp)$/i.test(fName);
  if (isImage) {
    messageText = text + '\n\n[用户上传了一张图片：' + fName + ']\n![图片](/api/download/' + encodeURIComponent(fName) + ')';
  } else {
    messageText = text + '\n\n[用户上传了文件：' + fName + ']';
  }
}
```

然后将 SSE 请求 URL 中的 `text` 替换为 `messageText`：

```javascript
const url = '/api/chat/stream?m=' + encodeURIComponent(messageText)
  + (currentAgentId ? '&agent=' + encodeURIComponent(currentAgentId) : '');
```

同时也要修改 `fallbackPost` 中的消息体，同样使用 `messageText`。

**llm.py 改动**：
检查 `handle_message` 函数中是否处理了图片 Markdown 语法。如果消息中包含 `![图片](/api/download/xxx.png)`，确保：
1. LLM 的上下文中能看到这个图片引用
2. 如果模型支持多模态，图片内容被正确编码发送

如果模型不支持多模态，至少确保 LLM 能看到文件名和用户描述。

## Bug 3：右侧面板定时频闪

### 现象
右侧面板每隔几秒闪烁一次，视觉上像刷新时的白屏/闪动。

### 根因分析
右侧面板每 10 秒调用 `renderRightPanel()` 刷新全部内容。如果刷新时先清空再渲染，或者网络请求慢导致内容短暂空白，就会产生频闪感。

### 修复方案

**app.js 改动**：

**方案 A（推荐）**：在 `renderRightPanel` 中不要清空整个面板再填充，而是只更新数据内容，不重建 DOM 结构。

找到 `renderRightPanel` 函数（约第 1022 行），修改为：

```javascript
function renderRightPanel() {
  // 不整体清空重建，而是逐个更新分区
  renderRightAgentList();
  renderRightTokenMeter();
  renderRightTimeline();
  renderRightMemory();
  renderRightBehavior();
}
```

同时修改 `renderRightAgentList` 等子函数，**不要用 `innerHTML = ''` 清空再赋值**，而是：
- 如果分区内容为空，首次渲染时设置内容
- 后续更新时，只更新数据值（如文本节点），不重建 DOM

或者在 `renderRightTimeline` 等函数中，添加防闪烁处理：

```javascript
function safeSetHTML(el, html) {
  if (!el) return;
  if (el.innerHTML === html) return;  // 内容没变就不更新
  el.innerHTML = html;
}
```

**方案 B**：增加刷新间隔，从 10 秒改为 30 秒。缺点是数据更新不及时。

**方案 C**：只在实际数据变化时才刷新。在每次刷新前比较新旧数据，无变化则跳过。

**推荐方案 A + C 结合**：
1. 子函数改为只更新文本值，不重建 DOM
2. 初次渲染时构建 DOM 结构，后续只更新 `.textContent`
3. 在 `renderRightTimeline` 等函数中缓存上一次的 HTML 字符串，相同则跳过

```javascript
// 在文件顶部添加缓存
let _lastTimelineHTML = '';

function renderRightTimeline() {
  // ...
  const html = '...';
  if (html === _lastTimelineHTML) return;  // 无变化，跳过
  _lastTimelineHTML = html;
  section.innerHTML = html;
}
```

对其他渲染函数（`renderRightAgentList`、`renderRightTokenMeter`、`renderRightMemory`、`renderRightBehavior`）也做同样的缓存处理。

## Bug 4：SSE 流式输出未恢复

### 现象
发送消息后，零一次性输出全部结果，没有逐字/逐段流式显示。

### 根因分析
003 任务包把 `handle_message` 迁移到 `run_in_executor` 后，可能在 executor 执行完成前 SSE 流就已经关闭，或者 executor 返回的结果没有通过流式通道逐块发送。

### 修复方案

**server.py 改动**：
检查 ChatStreamHandler 中的 executor 调用逻辑。确认以下几点：

1. 确保 `handle_message` 在 executor 中执行时，仍然能通过回调或队列向 SSE 流发送中间 chunk

2. 如果 `handle_message` 内部有 yield/chunk 回调，需要把回调函数也传入 executor：

```python
# 如果 handle_message 支持回调参数
async def streaming_chunk(data):
    await self._send({'type': 'chunk', 'data': data})

reply, agent = await loop.run_in_executor(
    None,
    lambda: handle_message(message, on_chunk=streaming_chunk)
)
```

3. 如果 `handle_message` 不支持回调，则在 executor 返回后，将完整结果一次性发送：

```python
# ensure data is sent as proper SSE chunk even when not streaming
if not self._chunk_sent:
    await self._send({'type': 'chunk', 'data': reply})
```

4. 确保 executor 的异常被正确捕获，不会静默吞掉：

```python
try:
    reply, agent = await loop.run_in_executor(None, handle_message, message)
except Exception as e:
    logger.error('handle_message executor failed: %s', e)
    await self._send({'type': 'error', 'data': str(e)})
    return
```

**核心检查点**：
- `handle_message` 的返回值是否通过 SSE 发送了 `chunk` 事件
- 如果 `handle_message` 本身是流式的（内部逐块产生输出），这些块是否被正确转发到 SSE
